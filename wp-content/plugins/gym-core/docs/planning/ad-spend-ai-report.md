## Why this matters

Existing CRMs report CAC and LTV separately. An ad agency report stops at CAC. Tying *ad-source cohort* → *retention curve* → *LTV* and recommending where to shift spend weekly is something no platform we surveyed offers. Once Phase 1 §F (lead-source at intake) has run for 6 months, we have the data to do it.

## Inputs

- Lead source per contact (Phase 1 §F).
- Subscription longevity per cohort (WC Subscriptions data, group by signup month + source).
- Ad spend per channel — manual import from Google Ads / Meta Ads CSV for v1; API integration in v2.
- Trial-to-member conversion rate per source (from funnel log in Phase 2 §J).

## Output (weekly)

- Per-channel scorecard: spend, leads, trials, members, 90-day retention, LTV-to-date, projected LTV.
- Recommendation: "Shift $X from Channel A to Channel B because Channel A's LTV cohort is decaying faster than CAC predicted." Generated by Finance agent + WP AI Client.
- Per-creative breakdown if creative ID was captured at lead intake (UTM-driven).

## Implementation notes

Per-plan:

- Module `src/AdReport/` in gym-core with `CohortBuilder/`, `Scorecard/`, `RecommenderAbility/`.
- Weekly Action Scheduler job emails the report to Darby + Andrew (HMA-internal distribution only — Latoya is at Automattic, not HMA, and is not in this loop).
- Recommender is a registered Ability so Gandalf can answer "should we keep running the Facebook campaign?" interactively.
- v1 imports CSVs; v2 wires Google Ads + Meta APIs. v1 is enough to be useful.

Implementation TODOs (expanded):

1. **Schema:**
   - `{$wpdb->prefix}gym_ad_spend` — `(week_start_date, channel, creative_id, spend_cents, impressions, clicks, source_csv_filename)`. Indexed on `(channel, week_start_date)`.
   - `{$wpdb->prefix}gym_ad_cohort` — derived view stored as a materialized table: `(cohort_month, source, creative_id, leads, trials, members, retention_90d, ltv_to_date_cents, projected_ltv_cents)`. Indexed on `(cohort_month, source)`.
2. **CSV import** (`src/AdReport/Import/CsvImporter.php`):
   - Admin page `Tools → Gym → Ad Spend Import` with two upload slots (Google Ads, Meta Ads).
   - Column-mapping config per channel (default profiles + override).
   - Idempotent on `(channel, week_start_date, creative_id)` — re-uploading the same week replaces, not duplicates.
3. **Cohort builder** (`src/AdReport/CohortBuilder/CohortBuilder.php`):
   - Joins `lead_source` (Phase 1 §F) → WC Subscription longevity → funnel log (§J trial conversion) → `gym_ad_spend`.
   - Persists into `gym_ad_cohort` weekly.
   - Exposes a query API for the scorecard renderer.
4. **Scorecard renderer** (`src/AdReport/Scorecard/`):
   - Per-channel + per-creative tables with the metrics listed in "Output".
   - Email-friendly HTML version for the weekly send.
   - Admin-page version under `WooCommerce → Reports → Ad Spend` mirroring brand-guide §6.
5. **Recommender** (`src/AdReport/Recommender/SpendShiftRecommender.php`):
   - Wraps the Finance agent (Phase 2 §H) with a structured prompt: input = scorecard + last 8 weeks of `gym_ad_cohort`, output = recommended shift list with rationale.
   - Confidence label on each recommendation; suppress recommendations below threshold.
6. **`recommend_ad_spend` Ability** in hma-ai-chat: Gandalf-callable wrapper around `SpendShiftRecommender` for interactive queries.
7. **Weekly job** (`src/AdReport/Jobs/WeeklyReportJob.php`):
   - Action Scheduler, Mondays 07:00 CT (after the visual-regression sweep).
   - Builds the report; emails the HTML version; archives the Markdown version under `wp-content/plugins/gym-core/docs/reviews/ad-spend/<date>.md` (git-tracked).
   - **Recipient list: Darby + Andrew. Hard-coded constants behind a filter; do not source from `wp_options` to avoid accidental expansion.**
8. **Trigger gate:** the report job must not run until 6 months of clean lead-source data exist. Implement an automatic gate that checks `lead_source` field-fill rate over the prior 6 months ≥ 90% before sending; below threshold = log + skip.
9. **Tests:** unit (cohort math, retention curve, projected-LTV formula); integration (CSV → cohort → scorecard end-to-end); recipient-list test (only `darby@... ` and `andrew@...` are sent to).
10. **CI gate:** `composer test-all`, PHPStan level 6, ESLint clean.
11. **v2 (API integrations):** Google Ads + Meta Ads OAuth; replaces the CSV importer for those channels but the table schema stays.

## Acceptance criteria

- First report ships exactly when lead-source data clears the 6-month mark.
- Darby (with Andrew as builder) can answer "what should we spend more on next month?" from the report alone.
- Spend-shift recommendations correlate with measured cohort improvement over the following quarter (validate continuously).

## Open questions

1. **LTV projection horizon.** Plan says "projected LTV" — over what horizon (12 months, 24 months, lifetime extrapolation)? Affects the model used (linear decay vs. cohort survival curve).
2. **CSV format normalization.** Google Ads and Meta Ads each have multiple export formats. Do we ship one canonical schema with column-mapping per source, or one importer-per-platform? v1 simplicity argues for the second; long-term flexibility argues for the first.
3. **Creative-ID capture.** UTM-driven creative tagging requires every ad URL to carry a stable `utm_content` value. Is the ad team disciplined enough today, or do we need a "fallback to channel-level" path for v1?
4. **Recommendation confidence floor.** What confidence threshold suppresses an AI recommendation? Below the floor: omit, or render with an "advisory" label?
5. **Recipient hard-coding vs. settings.** Plan says "Darby + Andrew" — store as code constants (safest, requires PR to change) or behind a `manage_options` settings page (more flexible, but accidental expansion risk)? Default to constants.
6. **Failure-mode behavior.** If the Finance agent times out or returns malformed JSON, does the weekly report still send (without recommendations) or fail closed? Default to "send scorecard, omit recommendations, log error".
