# Agent Accounts & MCP Registration — Implementation Plan

> **Status:** In Progress
> **Branch:** `feat/agent-accounts-and-mcp`
> **Date:** 2026-04-11
> **Author:** Andrew Wikel

## Context

Two features from the architecture doc (`ai-architecture-paperclip.md`) that have not been implemented:

1. **Per-agent WordPress user accounts** — so agent-authored content (social posts, announcements) has a clean audit trail tied to the agent persona, not the approving staff member.
2. **WordPress Abilities API / MCP registration** — so all 26 Gandalf tools are discoverable via the WP 7.0 MCP Adapter, making them accessible to any MCP-compatible client.

These features were validated against ClawPress (Newspack/aip2 WordPress plugin for AI agent site access). gym-core's existing architecture already covers the same ground; these additions close the remaining gaps.

---

## Feature 1: Per-Agent WordPress User Accounts

### Problem

When Gandalf drafts a social post or announcement, the `post_author` is set to whichever staff member was chatting. This makes the audit trail misleading — the content was generated by an agent, not the staff member.

### Solution

Create 4 dedicated WordPress user accounts (`gandalf-sales`, `gandalf-coaching`, `gandalf-finance`, `gandalf-admin`) with a custom `hma_ai_agent` role. These accounts:

- Cannot log in (authentication blocked via `authenticate` filter)
- Are hidden from the Users list table
- Own agent-authored content (`post_author` set to agent user on approved writes)
- Are preserved on plugin deactivation (audit trail integrity)
- Are deleted only on full uninstall

### Files

| Action | File | What Changes |
|--------|------|-------------|
| **Create** | `src/Agents/AgentUserManager.php` | New class: agent user lifecycle (create, sync caps, block login, lookup) |
| **Modify** | `src/Activator.php` | Call `AgentUserManager::provision()` on activation |
| **Modify** | `src/Plugin.php` | Hook login blocker, user list filter, ensure agents on init |
| **Modify** | `src/Tools/ToolExecutor.php` | Add `agent_user_id` to pending action data; use agent user for approved write execution |
| **Modify** | `uninstall.php` | Clean up agent users and custom role |

### Dual User Context

The key design: **staff member authorizes, agent user authors**.

- Authorization check: `user_can( $staff_user_id, $tool['auth_cap'] )` — unchanged
- Content authorship: `wp_set_current_user( $agent_user_id )` during write execution
- PendingAction stores both: `requested_by` (staff) and `agent_user_id` (agent)

### Edge Cases

- **Agent user manually deleted**: `get_agent_user_id()` detects missing user and re-creates
- **Plugin deactivated**: Agent users remain (inert — custom role, can't log in)
- **Plugin uninstalled**: Agent users and role deleted

## Feature 2: WordPress Abilities API / MCP Registration

### Problem

The 26 tools in `ToolRegistry` are only accessible through two paths: the Gandalf chat UI (`MessageEndpoint`) and Paperclip's heartbeat webhook. They can't be discovered or invoked by external MCP clients (e.g., Claude Desktop, other WP 7.0 agents).

### Solution

Register all tools as WordPress abilities via `wp_register_ability()` on the `wp_abilities_api_init` hook. Each ability:

- Maps to a `ToolExecutor::execute()` call (same dispatch pipeline)
- Uses the tool's existing `auth_cap` as the permission callback
- Preserves write gating (write tools still go through PendingAction approval)
- Is namespaced under `gandalf/` prefix (e.g., `gandalf/get-pricing`)

### Files

| Action | File | What Changes |
|--------|------|-------------|
| **Create** | `src/MCP/AbilitiesRegistrar.php` | New class: registers tools as WP abilities |
| **Modify** | `src/Plugin.php` | Hook abilities registration on `wp_abilities_api_init` |

### MCP Caller Behavior

- **Read tools**: Execute immediately, return data (same as internal chat dispatch)
- **Write tools**: Return `{pending: true, action_id: N}` — MCP caller must poll `ActionEndpoint` for approval status
- **Persona-agnostic**: MCP callers pass empty persona (skips persona access check); auth is capability-based only
